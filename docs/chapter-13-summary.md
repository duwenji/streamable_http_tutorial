# 章13 まとめ

## この章で振り返ること

- SSEとStreamable HTTPの本質的な違い
- 方式選定の判断フレーム
- 実務導入時に最初に確認すべき点

## 本質的な違い

### SSE

- サーバーからクライアントへの一方向通知に強い
- イベントフォーマットが定型で学習しやすい
- ブラウザ実装を短く保ちやすい

### Streamable HTTP

- メッセージ形式の自由度が高い
- 受信境界と再試行戦略を柔軟に設計できる
- その分、受信実装の責務が増える

## 方式選定フレーム

次の順で判断すると、選定理由を説明しやすくなります。

1. 通信方向
   - 一方向通知が中心ならSSEを優先
2. メッセージ設計自由度
   - 柔軟性が必要ならStreamable HTTPを検討
3. 実装コスト
   - 短期導入ならSSEが有利
4. 運用要件
   - 監視、復旧、ログ戦略を含めて判断

## 実務導入で最初に確認すべき点

- 更新頻度と許容遅延
- 同時接続規模
- 切断時の復旧方針
- 可観測性の要件
- セキュリティと認証要件

## 案Aでの推奨学習結論

進捗バー配信という題材では、次の流れが有効です。

1. SSEで最短実装を完成させる
2. Streamable HTTPで受信処理の設計力を強化する
3. 同一要件で比較し、選定理由を文章化する
