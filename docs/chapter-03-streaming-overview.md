# 章3 ストリーミング通信の全体像

## この章で学ぶこと

- Polling、Long Polling、SSE、Streamable HTTPの位置づけ
- 逐次配信が必要になる場面
- 方式選定の最初の判断軸

## 背景

リアルタイム表示を実装する場面では、
「更新をどう届けるか」を最初に決める必要があります。

代表的な方式は次の4つです。

- Polling
- Long Polling
- SSE
- Streamable HTTP

この章では、違いを俯瞰し、後続章の比較に備えます。

## 仕組み

### 1. Polling

クライアントが一定間隔で繰り返し問い合わせます。

- 利点: 実装が単純
- 欠点: 無駄なリクエストが増える

### 2. Long Polling

サーバーは更新があるまで応答を保留し、更新時に返します。

- 利点: Pollingより効率が良い
- 欠点: 接続管理と再要求の制御が必要

### 3. SSE

1本の接続でサーバーからイベントを継続配信します。

- 利点: 一方向通知に強く実装しやすい
- 欠点: イベント形式が固定寄り

### 4. Streamable HTTP

通常HTTPレスポンスを逐次送信し、クライアントが順次読み取ります。

- 利点: 形式設計の自由度が高い
- 欠点: 受信処理の責務が増える

## 比較マップ

### 通信効率

- 低: Polling
- 中: Long Polling
- 高: SSE / Streamable HTTP

### 実装難易度

- 低: Polling
- 中: SSE
- 中から高: Streamable HTTP, Long Polling

### 受信処理の自由度

- 低: SSE
- 高: Streamable HTTP

## 案Aへの適用

進捗バー配信の題材では、次の順で学ぶと理解しやすいです。

1. SSEで最短実装を体験する
2. Streamable HTTPで境界処理を学ぶ
3. 同一要件で比較して選定根拠を作る

## 最小選定フロー

1. 一方向通知が中心か
2. 受信形式の自由度が必要か
3. 短期間での実装が必要か
4. 障害時の復旧設計をどこまで自前で持つか

この4点で第一候補を決め、詳細は章10で詰めます。

## よくある失敗

- Pollingとストリーミングの運用コスト差を過小評価する
- 方式選定を実装者の慣れだけで決める
- 異常系検証を行わず、比較が主観になる
- 監視指標を決めずに実装を先行する
