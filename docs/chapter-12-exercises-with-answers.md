# 章12 演習問題と解答例

## 使い方

- まずは解答を見ずに取り組む
- 次に解答例で観点の漏れを確認する
- 最後に自分の案件条件へ置き換えて書き直す

## 基礎問題

### 問1

進捗バー配信にSSEを使うとき、最低限必要なレスポンスヘッダーを挙げる。

### 解答例

- Content-Type: text/event-stream
- Cache-Control: no-cache
- Connection: keep-alive

### 問2

Streamable HTTPでJSON Linesを採用する理由を説明する。

### 解答例

1行を1メッセージ境界にできるため、受信側が分割到着を安全に復元しやすい。さらにログ可読性が高く、障害時の切り分けがしやすい。

## 設計問題

### 問3

1万人同時接続を想定した場合、どの観測指標を優先監視するか。

### 解答例

- アクティブ接続数
- 切断率
- 再接続回数
- 配信遅延
- エラー率

優先理由は、接続増加に伴う劣化を最速で検知できるため。

### 問4

途中切断を前提としたとき、クライアント復旧戦略を設計する。

### 解答例

- 再試行は指数バックオフ
- 最大試行回数を設定
- 最終処理済み進捗を保持
- 再接続後は重複反映を抑止
- 上限超過時は失敗表示と再実行導線を提示

## 比較問題

### 問5

同じ要件でSSEとStreamable HTTPを比較し、採用理由を200字で述べる。

### 解答例

進捗バー配信ではSSEを採用する。理由は、ブラウザ標準APIで実装が短く、イベント形式が明快で初期学習コストが低いため。再接続挙動も整理しやすく、教材目的である通信理解に集中できる。将来、受信形式の自由度が必要になった場合はStreamable HTTPへ拡張する。

### 問6

デバッグ容易性を上げるためのログ設計を両方式で提案する。

### 解答例

- SSE
  - 送信時にevent名、id、jobId、percent、送信時刻を記録
  - 受信時にevent名、反映結果、再接続回数を記録
- Streamable HTTP
  - 行確定時にtype、jobId、percent、parse成否を記録
  - バッファ長、タイムアウト、再試行回数を記録

## 章末チェック

次を満たせば到達条件を満たします。

- 解答例を丸写しせず、自分の要件で説明できる
- 方式選定と運用設計を一体で説明できる
- 障害時の初動を3手以内で提示できる
