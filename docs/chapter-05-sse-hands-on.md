# 章5 SSEハンズオン

## この章で学ぶこと

- 進捗バー配信をSSEで実装する手順
- イベント設計の最小ルール
- 接続切断と再接続の確認方法
- 失敗シナリオを使った耐性確認

## 背景

SSE基礎を理解しても、実装すると次の壁に当たりやすいです。

- どのイベントを定義すべきか
- 何をデータに含めるべきか
- どの時点で完了扱いにするか
- 途中切断をどう扱うか

この章では、案Aの進捗バー配信を題材に、実装から検証までを一気に通します。

## 実装前の決めごと

### エンドポイント

- GET /progress-sse

### イベント設計

- progress: 進捗更新
- done: 完了通知
- error: 業務上の失敗通知（任意）

### データスキーマ

progressイベントのdataは次を必須にします。

- jobId: 文字列
- percent: 0から100の整数
- message: 画面表示用の短文
- timestamp: ISO 8601文字列

## 最小サーバー実装の流れ

1. SSEヘッダーを返す
2. 進捗0から100まで一定間隔で更新する
3. 各更新でprogressイベントを送る
4. 最後にdoneイベントを送る
5. 接続クローズを行う

サーバー送信イメージは次の通りです。

```text
event: progress
data: {"jobId":"job-1","percent":0,"message":"start","timestamp":"..."}

...

event: progress
data: {"jobId":"job-1","percent":100,"message":"complete","timestamp":"..."}


event: done
data: {"jobId":"job-1"}

```

## 最小クライアント実装の流れ

1. EventSourceで接続する
2. progressを受けたら進捗バー値を更新する
3. doneを受けたら完了表示へ切り替える
4. エラー時の表示と再接続待機状態を示す

## 検証手順

### 正常系

1. 接続開始後、0から100へ単調増加することを確認
2. 0、25、50、75、100で表示が崩れないことを確認
3. done受信時にボタン状態や表示文言が完了へ変わることを確認

### 異常系

1. サーバーを途中停止してUIがクラッシュしないことを確認
2. 再起動後に再接続できることを確認
3. 不正JSONを1件送っても以降のイベント処理が継続することを確認

## 実装のポイント

- percentは必ず単調増加で扱う
- 同じpercentが再送される可能性を考慮し、UI更新は冪等にする
- doneを受信したら二重完了処理を防ぐ

## よくある失敗

- progressイベント名のタイポでUI更新されない
- JSON parse失敗時に全体処理を停止してしまう
- ネットワークエラー時のユーザー通知がない
- doneとcloseの順序が不安定で完了表示が漏れる

## 章末問題

1. progressイベントにtimestampを入れる理由を説明してください。
2. 再接続後の重複更新を抑えるクライアント戦略を提案してください。
3. doneイベントを用意しない設計の問題点を挙げてください。
