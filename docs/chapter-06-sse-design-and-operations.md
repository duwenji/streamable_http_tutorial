# 章6 SSEの設計と運用

## この章で学ぶこと

- SSEを本番運用する際の設計観点
- 接続維持、心拍、再接続の考え方
- 認証とセキュリティの注意点
- 監視指標と障害切り分け手順

## 背景

SSEは実装自体は比較的シンプルですが、運用段階では次の問題が発生しやすいです。

- 接続が途中で切れる
- 一部環境でイベント遅延が増える
- 同時接続が増えるとサーバー負荷が急増する
- 障害時に原因が特定しづらい

この章では、案Aの進捗バー配信を対象に、運用で破綻しにくい設計を整理します。

## 設計

### 1. 一方向通信を前提にする

SSEはサーバーからクライアントへの通知に最適化されています。

- 上り方向の操作は通常HTTP APIへ分離
- SSEは通知専用チャネルとして扱う

### 2. イベント種別と責務を分離する

- progress: 進捗更新
- done: 正常完了
- error: 業務エラー
- heartbeat: 接続維持用コメント

責務を分離すると、クライアント側処理が明確になります。

### 3. 再接続を前提にイベントIDを設計する

- idには単調増加の連番を使う
- クライアントは再接続時に重複反映を防ぐ
- 最後に処理したidを保持し、逆行更新を無視する

### 4. 心拍を定期送信する

アイドル時間が長いと中間機器で切断されやすくなります。

- コメント行を一定間隔で送る
- 心拍受信の間隔異常を監視対象にする

## 運用

### 認証

- アクセストークンは短寿命にする
- 接続開始時の認可チェックを必須化する
- 権限が切れた場合の切断と再認可導線を明確にする

### 負荷対策

- 同時接続数の上限を設ける
- 接続1本あたりのメモリ使用量を可視化する
- 高負荷時に更新間隔を自動調整する

### 監視指標

最低限、次を監視します。

- アクティブ接続数
- 接続継続時間
- 切断率
- 再接続回数
- イベント配信遅延
- 送信失敗率

### 障害切り分け

1. 接続確立失敗か、途中切断かを判定する
2. サーバーログと中間機器ログを突き合わせる
3. 心拍間隔の乱れを確認する
4. クライアント側の再接続挙動を確認する

## よくある失敗

- 心拍未実装で一部経路のみ切断が多発する
- すべての失敗をerrorイベントで返し、運用分析が難しくなる
- 接続上限を決めず、ピーク時にサーバーが不安定になる
- 再接続時の重複反映で進捗バーが逆行する

## 章末問題

1. SSEでイベントIDを導入する運用上の利点を説明してください。
2. 心拍間隔を監視指標に含める理由を説明してください。
3. 同時接続急増時の段階的な緩和策を提案してください。
