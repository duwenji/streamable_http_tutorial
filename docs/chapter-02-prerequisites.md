# 章2 前提知識

## この章で学ぶこと

- HTTPの基本的な通信モデル
- Keep-Aliveと接続維持の考え方
- チャンク転送の意味
- Content-Typeの役割
- ストリーミングで重要なタイムアウト設計

## 背景

SSEとStreamable HTTPは、どちらもHTTPの上で成り立ちます。
そのため、HTTPの基本を曖昧なまま進むと、次のような誤解が起きやすくなります。

- なぜレスポンスがすぐ終わらないのか
- なぜ途中で切断されるのか
- なぜ受信側で境界処理が必要なのか

この章では、後続の実装でつまずきやすい論点だけを絞って整理します。

## 仕組み

### 1. HTTPの基本

HTTPは通常、次の流れで1往復します。

1. クライアントがリクエストを送る
2. サーバーがレスポンスを返す
3. 処理が完了して接続を閉じるか再利用する

通常のAPIではレスポンスをまとめて返しますが、
ストリーミングではレスポンスを小分けで返し続けます。

### 2. Keep-Alive

Keep-Aliveは接続をすぐ閉じずに維持し、再利用する仕組みです。

- 接続確立のオーバーヘッドを減らせる
- ただしアイドル時間が長いと中間機器に切られる場合がある

### 3. チャンク転送

レスポンス全体サイズが事前に確定しなくても、
データを分割して順次送れる仕組みです。

重要なのは、転送上の分割単位とアプリの意味単位は一致しないことです。

### 4. Content-Type

受信側はContent-Typeをもとに解釈方法を決めます。

- SSE: text/event-stream
- Streamable HTTPの例: application/x-ndjson

### 5. タイムアウト

タイムアウトは次の層で起きます。

- クライアント
- サーバー
- プロキシやロードバランサ

どこで切れるかを把握しないと、再現性の低い障害になります。

## 最小例

### SSEのレスポンスヘッダー例

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

### Streamable HTTPのレスポンスヘッダー例

```http
HTTP/1.1 200 OK
Content-Type: application/x-ndjson
Cache-Control: no-cache
Connection: keep-alive
Transfer-Encoding: chunked
```

## 実装で意識するポイント

- 接続が長く続く前提で設計する
- 境界処理を必ず実装する
- タイムアウトと再試行をセットで考える
- ログに時刻と接続状態を残す

## よくある失敗

- チャンク分割をそのまま1メッセージと誤解する
- Content-Typeを曖昧にしてクライアント解釈が壊れる
- タイムアウト値を未設定にして本番だけ切断多発する
- 接続維持中の監視指標を持たない

## 章末問題

1. Keep-Aliveがストリーミングで重要な理由を説明してください。
2. チャンク転送とメッセージ境界を区別して説明してください。
3. タイムアウトが多層で発生することによる運用上の難しさを説明してください。
