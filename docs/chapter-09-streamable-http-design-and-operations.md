# 章9 Streamable HTTPの設計と運用

## この章で学ぶこと

- Streamable HTTPの本番設計で重要な観点
- メッセージ境界とバッファ設計
- 途中切断と再試行戦略
- 監視とデバッグの実践ポイント

## 背景

Streamable HTTPは柔軟性が高い反面、実装責務が増えます。

- 受信境界の設計
- 失敗時の再試行
- 完了判定
- ログ設計

この章では、案Aの進捗バー配信で運用可能な設計へ落とし込みます。

## 設計

### 1. メッセージ境界を明文化する

本教材ではJSON Linesを採用します。

- 1行1メッセージ
- 行末改行で境界確定
- typeで処理分岐

### 2. 受信バッファを安全に扱う

- チャンク単位ではparseしない
- 完全行のみparseする
- 末尾不完全行は次チャンクまで保持する
- バッファ上限を決めて防御する

### 3. 完了条件を明確化する

- type=doneを受信したら正常終了
- 一定時間無通信ならタイムアウト
- タイムアウト時は再試行か失敗表示へ遷移

### 4. 再試行戦略を設計する

- 指数バックオフを採用する
- 最大試行回数を決める
- 再試行後の重複反映を抑制する

## 運用

### ログ設計

- 送信側: 行送信時刻、type、jobId、percent
- 受信側: 行確定時刻、parse成否、UI反映結果
- エラー: 失敗種別、直前行、再試行回数

### 監視指標

- ストリーム開始率
- 正常完了率
- タイムアウト率
- parse失敗率
- 平均完了時間
- 再試行成功率

### 障害切り分け

1. サーバー送信が止まっているか確認
2. ネットワーク分割到着でパース失敗していないか確認
3. done未到達か、done処理漏れかを判定
4. 再試行回数と成功率の相関を確認

## よくある失敗

- JSON Lines境界を守らず受信ロジックが複雑化する
- parse失敗を致命扱いにして全体停止する
- doneが欠落したときの終了戦略がない
- タイムアウト値が環境と不整合で誤検知が多発する
