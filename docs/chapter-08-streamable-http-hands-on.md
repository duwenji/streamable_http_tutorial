# 章8 Streamable HTTPハンズオン

## この章で学ぶこと

- 進捗バー配信をStreamable HTTPで実装する手順
- JSON Lines受信ループの作り方
- 途中切断や分割到着への対処
- 完了判定とUI遷移の実装

## 背景

基礎理解だけでは、実装時に次の課題が残ります。

- どこでメッセージを確定するか
- 不完全なデータをどう保持するか
- いつ読み取りを終了するか

この章では案Aを使い、実装を段階化して確実に完成させます。

## 実装前の決めごと

### エンドポイント

- GET /progress-stream

### データ形式

- 1行1JSONのJSON Lines
- typeはprogressまたはdone

### 共通フィールド

- type
- jobId
- percent
- message
- timestamp

## サーバー実装の流れ

1. 逐次送信レスポンスを開始する
2. 進捗0から100を一定間隔で生成する
3. 各更新をJSON Linesで1行送る
4. 最後にtype=doneを送る
5. レスポンスを終了する

送信イメージは次の通りです。

```text
{"type":"progress","jobId":"job-1","percent":0,"message":"start","timestamp":"..."}
{"type":"progress","jobId":"job-1","percent":25,"message":"working","timestamp":"..."}
...
{"type":"done","jobId":"job-1","percent":100,"message":"complete","timestamp":"..."}
```

## クライアント実装の流れ

1. ストリーム読み取りを開始する
2. チャンクを文字列へ変換してバッファへ追記
3. 改行単位で完全行を取り出す
4. 各行をJSON parseしてtypeごとに処理する
5. done受信でループ終了しUIを完了状態へ遷移

## 検証手順

### 正常系

1. 0から100へ進捗が単調増加する
2. 0、25、50、75、100を確認できる
3. done受信で読み取りが終了する

### 異常系

1. サーバー停止時にUIが異常終了しない
2. 分割到着行でも次チャンクで復元できる
3. 不正行が混ざっても残り処理を継続できる

## 実装のポイント

- 行境界を基準にparseする
- parse失敗を致命扱いにしない
- doneを単一責務で扱い終了条件を明確化する
- 進捗値は単調増加のみ反映し、逆行更新を無視する

## よくある失敗

- 受信チャンク単位でJSON parseしてしまう
- バッファ末尾の不完全行を捨てて情報欠落する
- done未受信時のタイムアウト処理がない
- 例外時にローディング表示が残る

## 章末問題

1. 分割到着を前提にした受信ループの擬似コードを書いてください。
2. doneが届かない場合の終了戦略を2案示してください。
3. 不正行を検知したときのUIとログの方針を提案してください。
