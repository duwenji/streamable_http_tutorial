# 章7 Streamable HTTP基礎

## この章で学ぶこと

- Streamable HTTPの基本モデル
- chunked transferとアプリ層メッセージ境界の違い
- JSON Linesで逐次配信する考え方
- 進捗バー配信での適用ポイント

## 背景

SSEは便利ですが、イベントフォーマットが固定です。
一方、Streamable HTTPは通常のHTTPレスポンスを逐次返すため、
メッセージ形式を用途に合わせて設計しやすい利点があります。

進捗バー配信では、次のニーズで有効です。

- 既存HTTP基盤へ組み込みたい
- ペイロード形式を自由に決めたい
- SSE専用APIではなく統一的な受信処理を使いたい

## 仕組み

クライアントは通常のGETを送信し、サーバーはレスポンスを一括で返さず、
小さな単位で順次書き出します。クライアントは受信した断片を逐次解釈します。

基本フローは次の通りです。

1. クライアントがGET /progress-streamへ接続
2. サーバーが逐次送信可能なレスポンスを開始
3. サーバーが進捗データを小分けで送る
4. クライアントが読み取りループで順次処理
5. doneを受信して処理を終了

### 重要な区別

- 転送層の分割: ネットワーク都合で任意に分割される
- アプリ層のメッセージ境界: 受信側が意味単位として扱う区切り

この2つを混同すると、パース不具合を起こしやすくなります。

## 最小例

### JSON Lines形式の例

```text
{"type":"progress","jobId":"job-1","percent":0,"message":"start","timestamp":"..."}
{"type":"progress","jobId":"job-1","percent":25,"message":"working","timestamp":"..."}
{"type":"done","jobId":"job-1","percent":100,"message":"complete","timestamp":"..."}
```

1行が1メッセージになるため、境界判定が単純です。

### クライアント受信の最小イメージ

- 受信バッファへ追記
- 改行で分割
- 完全な1行だけJSON parse
- 不完全な末尾は次チャンクまで保持

## 実装

案Aの進捗バー配信では、次の方針を採用します。

- エンドポイント: GET /progress-stream
- 形式: JSON Lines
- メッセージタイプ: progress, done
- 共通フィールド: type, jobId, percent, message, timestamp

## よくある失敗

- チャンク単位でJSON parseし、分割到着時に失敗する
- done判定が曖昧で読み取りループが終わらない
- バッファが肥大化してメモリ使用が増える
- 送信側がflushしないため体感遅延が大きくなる
