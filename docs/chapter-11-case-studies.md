# 章11 ケーススタディ

## この章で学ぶこと

- ユースケースごとの方式選定
- 実装時の優先論点
- 運用時に観測すべき指標

## ケース1 進捗通知

### 要件

- 更新頻度は中程度
- 画面でリアルタイム表示したい
- 実装を短期間で立ち上げたい

### 推奨

- 第一候補: SSE

### 理由

- ブラウザで扱いやすい
- イベントモデルが明快
- 実装学習コストが低い

## ケース2 ライブログ表示

### 要件

- 連続的にログを受けたい
- 長時間接続が前提
- 障害時に復旧しやすくしたい

### 推奨

- 第一候補: SSE
- 条件次第でStreamable HTTP

### 理由

- SSEは接続継続とイベント処理がシンプル
- ただしメッセージ形式の自由度が必要ならStreamable HTTPが有利

## ケース3 LLM応答の逐次表示

### 要件

- トークン単位または文単位で逐次表示
- クライアント受信ロジックを最適化したい
- 将来的な形式拡張があり得る

### 推奨

- 第一候補: Streamable HTTP

### 理由

- ペイロード形式を柔軟に設計できる
- 受信処理を用途ごとに最適化しやすい

## ケース4 通知配信

### 要件

- 単方向通知
- ステータス更新を素早く反映
- 大規模同時接続も想定

### 推奨

- 第一候補: SSE

### 理由

- 単方向通知に適合
- 設計がシンプルで運用しやすい

## ケース別チェックポイント

- 通信方向は一方向か
- メッセージ形式の自由度は必要か
- 受信処理の複雑化を許容できるか
- 障害復旧の戦略をどこまで自前実装するか

## 章末問題

1. あなたの想定案件を1つ選び、方式選定理由をまとめてください。
2. 選定した方式で運用監視すべき指標を5つ挙げてください。
3. 採用しなかった方式のリスクも説明してください。

## 章末チェック

次を説明できれば到達条件を満たします。

- 代表ユースケースでの方式選定
- 要件から見たトレードオフ
- 運用時の監視観点
