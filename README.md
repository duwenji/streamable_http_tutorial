# SSE and Streamable HTTP Tutorial

## 目的

SSEとStreamable HTTPを、HTTPストリーミングの基礎から比較・実装・設計判断まで一貫して学ぶ教材を作る。

この教材ではMCPは主題にしない。まずは通信方式そのものを理解し、どの要件でどちらを選ぶべきか判断できる状態を目指す。

## 想定読者

- HTTPの基本は知っているが、ストリーミング通信の実装経験は少ない人
- SSEとStreamable HTTPの違いを整理して理解したい人
- LLM応答表示、進捗通知、ライブログ表示のようなUIを実装したい人

## 到達目標

- SSEの通信モデルと基本実装を説明できる
- Streamable HTTPの通信モデルと基本実装を説明できる
- 両者の違いを、実装・運用・ユースケースの観点で比較できる
- 要件に応じて、どちらを採用すべきか判断できる

1. 全体構成を決める

## 第4段階: SSE版ハンズオン要件（案A）

### 学習シナリオ

- 疑似ジョブを0から100まで進捗更新する
- サーバーは1秒ごとに進捗イベントを送る
- クライアントは受信するたびに進捗バーを更新する

### 実装要件

- エンドポイント
	- GET /progress-sse
- レスポンス
	- Content-Typeはtext/event-stream
	- 心拍コメントを定期送信する
	- 進捗イベントはevent: progressで送る
	- 完了イベントはevent: doneで送る
- データ形式
	- progressイベントはJSONで送る
	- フィールドはjobId、percent、message、timestamp

### 検証要件

- 0、25、50、75、100の更新がUIで確認できる
- 接続切断後に再接続して受信を再開できる
- done受信時にUIが完了状態へ遷移する

### 失敗シナリオ

- サーバーを途中停止して再接続挙動を確認する
- 不正データを1回だけ送ってクライアントの耐性を確認する

## 第5段階: Streamable HTTP版ハンズオン要件（案A）

### 学習シナリオ

- SSE版と同じ疑似ジョブを逐次レスポンスで配信する
- クライアントはストリームを順次読み取り、進捗バーを更新する

### 実装要件

- エンドポイント
	- GET /progress-stream
- レスポンス
	- 逐次送信可能なHTTPレスポンス
	- JSON Lines形式で1行ずつ送る
	- 完了行はtype=doneで送る
- データ形式
	- 1行ごとのJSONにtype、jobId、percent、message、timestampを含める

### 検証要件

- 逐次更新がSSE版と同等に見える
- 受信バッファの切れ目があっても正しく復元できる
- done受信時に読み取りループを正しく終了できる

### 失敗シナリオ

- 途中切断時にUIが異常終了しない
- 1行が分割到着してもパースできる

### 実験手順

1. 共通の進捗UIを用意する
2. SSE版を実装して動作確認する
3. Streamable HTTP版を実装して動作確認する

5. 異常系テストを両方式で実行する
6. 観測ログを記録する
7. 評価シートを埋める

### 正常系テスト項目

- 進捗が0から100へ単調増加する
- 0、25、50、75、100の節目表示が一致する
- 完了時にUIが完了状態へ遷移する
- 体感の更新遅延が許容範囲である

### 異常系テスト項目

- サーバー途中停止時にUIがクラッシュしない
- 再接続または再読込後に復旧できる
- 不正フォーマット1件を受けても処理継続できる
- 分割到着データを正しく復元できる
